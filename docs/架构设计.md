# 青苗守护者 · 架构设计

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      浏览器 (Vue 3 SPA)                          │
│  数据大屏 / 管理后台 / 课堂分析 / 心理档案 / AI 对话 / 量表       │
└────────────┬───────────────────────────────────┬────────────────┘
             │ HTTPS REST                        │ WebSocket
             ↓                                   ↓
┌─────────────────────────────────────────────────────────────────┐
│                  Flask 后端 (gevent + SocketIO)                  │
│  用户/权限 RBAC | 业务逻辑 Service | Celery 队列 | WS 推送       │
└─────┬─────────────┬─────────────┬─────────────┬─────────────────┘
      ↓             ↓             ↓             ↓
   MySQL         Redis         MinIO       AI 推理服务
   业务数据     缓存/队列      视频/文件   (FastAPI 微服务)
                                                ↓
                                  本地模型 + 通义千问 API
```

## 二、模块划分

### 后端（Flask）

- `app/api/`：REST 蓝图集合（每个领域一个文件）
- `app/models/`：SQLAlchemy ORM 模型
- `app/schemas/`：Marshmallow 序列化与请求校验
- `app/services/`：业务逻辑层，负责事务与跨表逻辑
- `app/ai/`：AI 客户端封装（调用 ai_service 微服务、通义千问 SDK）
- `app/tasks/`：Celery 异步任务（视频分析、报告生成等）
- `app/sockets/`：WebSocket 事件（实时摄像头、任务进度推送）
- `app/utils/`：通用工具

### AI 推理服务（FastAPI）

- `pipelines/behavior_pipeline.py`：YOLOv8 + DeepSORT，行为检测+追踪
- `pipelines/emotion_pipeline.py`：HSEmotion，表情识别
- `pipelines/face_pipeline.py`：InsightFace，人脸检测+识别
- `pipelines/text_pipeline.py`：通义千问，文本情绪/风险分析
- `server.py`：FastAPI 入口，对外暴露 REST 接口供 Flask 调用

为什么独立成微服务：
1. AI 依赖（PyTorch、CUDA）很重，与业务后端解耦，避免拖累后端启动
2. 可独立水平扩展（多实例 + 队列分发）
3. 可独立用 GPU 镜像，业务后端用普通镜像

### 前端（Vue 3）

- `views/dashboard/`：DataV 风格数据大屏
- `views/admin/`：用户、组织架构、角色权限管理
- `views/classroom/`：课堂视频上传与分析、实时摄像头
- `views/psychology/`：心理量表、文本分析、AI 对话
- `views/correlation/`：多维关联分析
- `views/alerts/`：预警中心、工单流转
- `views/reports/`：报告查看与导出
- `views/login/`：登录

## 三、数据流

### 1. 上传视频分析流程

```
用户上传视频 → 后端 (Flask) → MinIO 存储 → 创建分析任务 (DB)
   ↓
Celery Worker 拉取任务 → 调用 AI 服务 (HTTP) → 接收分析结果
   ↓
写入 DB (behavior_records / emotion_records) → 通过 WS 推送进度给前端
   ↓
前端展示分析报告（视频回放 + 多维图表）
```

### 2. 实时摄像头分析流程

```
浏览器 WebRTC 采集 → 抽帧 → WebSocket 推送 base64 帧
   ↓
Flask SocketIO 接收 → 转发到 AI 服务 (HTTP) → 推理
   ↓
推理结果通过 WebSocket 实时推回前端 → Canvas 叠加渲染框/标签
```

### 3. 心理量表流程

```
教师选择学生 → 学生作答 → 后端自动评分 → 写入 scale_assessments
   ↓
若评分超过阈值 → 创建预警工单 → 站内消息 + 邮件通知心理老师
```

### 4. 关联分析流程

```
后端定时任务 → 拉取学生多维数据 (成绩 + 行为 + 表情 + 量表)
   ↓
计算相关系数 / 异常评分 / 综合风险等级
   ↓
更新预警状态 → 大屏 / Dashboard 实时刷新
```

## 四、权限模型（M1 详细设计）

| 角色 | 数据范围 | 特殊权限 |
|---|---|---|
| 学校管理员 | 本校全部 | 用户管理、配置 |
| 心理学老师 | 本校全部 | 同管理员 + 预警工单处理 |
| 年级组长 | 本年级 | 年级报告 |
| 班主任 | 本班全部科目 + 自任教科目跨班 | 班级报告 |
| 科任老师 | 自任教的「班级×科目」交叉范围 | 课堂分析 |

实现方式：基于（学校、年级、班级、科目）四元组做查询过滤，权限规则在 Service 层统一拦截。

## 五、部署拓扑

### 开发机（GPU P100）

```
docker compose up -d   # 全套启动
GPU 用于 AI 推理服务（开启 cuda）
```

### 演示机（无 GPU）

```
AI_DEVICE=cpu          # 强制 CPU 推理
模型用 ONNX CPU 版本
预生成部分演示数据    # 演示视频走预生成结果，避免现场卡顿
```
